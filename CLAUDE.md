# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

EduConnect is a private, invite-only one-on-one tutoring platform (one teacher, one student). The full product spec lives in `EduConnect_PRD.md` — consult it for feature requirements, RBAC matrix, and the canonical DB schema before adding features.

Adds `current_cohort_id()` SECURITY DEFINER helper (mirrors `is_teacher()`, used by cohort-scoped policies). Creates private storage buckets `masterclass-materials`, `masterclass-submissions`, `certificates` with RLS. Adds `handle_new_user()` trigger on `auth.users` that auto-provisions a `student_group` profile **only** for `provider='google'` sign-ups with no existing profile (email/admin-created Mode A accounts untouched); EXECUTE on it is revoked from anon/authenticated (trigger-only). **Role note:** PRD's `student_private` is mapped onto the existing `role='student'` value to avoid breaking ~20 Mode A call sites and all working RLS; `student_group` is the only genuinely new role.
  - `20260630000001_prompt_arena.sql` — **Prompt Quest (the "Prompt Arena")**, a game-style non-MCQ challenge testing the Week-2 5-ingredient prompt recipe. Adds `arena_topics` (seeded scenario pool, 18 per round × easy/medium/hard, keyed by `slug` for re-runnable seeds) and `arena_attempts` (one row per `(user_id, round)`, holds the claimed topic, the prompt, and the AI grade: `score` /10 + `breakdown` jsonb of five 0-2 sub-scores + feedback/strengths/fixes). A partial UNIQUE index `(cohort_id, round, topic_id)` guarantees **no two learners in a cohort share a scenario in the same round**. RLS: teacher full CRUD; learner SELECT own only — **no learner INSERT/UPDATE**, so a learner can never set their own score or peek at unseen scenarios. SECURITY DEFINER RPCs: `arena_state()` (per-learner board: 3 rounds with lock state + grade; scenarios for locked rounds withheld), `claim_arena_topic(p_round)` (claim/resume a unique scenario under a `pg_advisory_xact_lock` per cohort+round; enforces the easy→medium→hard gate — previous round must be `graded`). Also `CREATE OR REPLACE`s `masterclass_leaderboard()` (same return signature) to **fold arena points into `total_points`/`possible_points`** so quiz + quest XP share one board.
- `supabase/functions/grade_prompt/` — Deno Edge Function: the **AI judge** behind Prompt Quest. Given `{ attempt_id, prompt }`, validates the caller owns the attempt (user JWT, `verify_jwt=true`), saves the prompt as `submitted` first (never lost on an LLM hiccup), then calls **Google Gemini** (same model-fallback + `responseSchema` pattern as `ingest_feeds`, low temp) to score the prompt 0-2 on each of WHO/WHAT/DETAILS/OUTPUT/STYLE, and writes `score`/`breakdown`/feedback as `graded` with the **service-role key** (so the score is unforgeable). Idempotent once graded; a failed grade stays `submitted` and is retryable. Requires `GEMINI_API_KEY` (shared with `ingest_feeds`). `[functions.grade_prompt] verify_jwt = true` in `config.toml`.
- `supabase/functions/ingest_feeds/` — Deno Edge Function that fetches each active `feed_sources.rss_url`, dedups by `source_url`, extracts cover images (RSS enclosure → og:image → first article `<img>`), proxies them into `branding/feed-covers/`, and calls **Google Gemini 2.5 Flash** (`responseSchema` mode) to rewrite the title + excerpt into student-friendly wording before inserting into `feed_items` with `status = 'published'`. Requires `GEMINI_API_KEY` secret. **Never review queue** — news is live by default; teacher only hides/pins/re-simplifies after the fact.
- `supabase/config.toml` — Supabase project config. The `[auth.email.template.recovery]` block points at `supabase/templates/recovery.html` (custom branded reset-password email).
- `supabase/templates/recovery.html` — Public-Sans + `#00193c → #002d62` academic-gradient password reset email. Logo `<img>` hot-links the Supabase public bucket URL (`branding/hat.png`); email clients can't load localhost or signed URLs.
- `EduConnect_PRD.md` — product requirements (features, schema, NFRs).

## Architecture

### Auth + routing flow
`src/App.tsx` is the single source of routing truth. On mount it calls `supabase.auth.getSession()` and subscribes to `onAuthStateChange`. Whenever a session exists, it fetches the user's `profiles` row and stores it in Zustand (`useAppStore`). The store derives `role` from `profile.role`. Unauthenticated users are redirected to `/login`; authenticated users hitting `/login` bounce to their role's dashboard.

The route guards (`teacherGuard` / `studentGuard` / `groupGuard`) have a three-state ladder: **no user → redirect to `/login`**; **user but role still loading → render `<AppShellSkeleton />`**; **user with wrong role → redirect to the correct home**. The middle state is critical — without it, the post-login transition produced a blank page because guards bounced through `/login` ↔ `/dashboard` while the profile fetch was in flight. `/login` itself uses the same ladder when `user` is set.

**Teacher mode switcher (Phase 7):** A teacher manages both modes from one panel via `components/TeacherModeSwitcher.tsx` (a "1:1 Track / Masterclass" segmented control at the top of the sidebar, teacher-only). The active mode is **derived from the path** (`/teacher/masterclass*` → Mode B), and `DashboardLayout` overrides the teacher's visible nav via `teacherNavForPath(pathname)` (→ `teacherNav` for Mode A, `teacherMasterclassNav` for Mode B) — so individual teacher pages don't pass mode-specific navs (the `navItems` they pass is ignored for teachers). The masterclass "Overview" hub uses exact-match (`end`) in the sidebar so it doesn't stay highlighted on sub-pages. Mode A and Mode B stay cleanly separated at the data layer too: Mode A admin queries (`Students.tsx`, `notify.ts`) filter `role='student'`, so they never surface `student_group` users, and vice-versa.

There are **three roles**: `teacher`, `student` (Mode A private DTU student = PRD's `student_private`), and `student_group` (Mode B masterclass cousin, Google OAuth). Wrong-role redirects (and the `/login` bounce) all go through `roleHome(role)` in `src/lib/nav.ts` — teacher→`/teacher/dashboard`, student→`/student/dashboard`, student_group→`/masterclass`. **Always use `roleHome()` rather than a hand-written `role === 'teacher' ? … : …` ternary**, because a two-way ternary will cross-bounce a `student_group` user into an infinite `/student` ↔ `/teacher` redirect loop. Mode B pages live under `/masterclass/*` (only the hub `/masterclass` exists so far — Phase 2 adds sessions/quizzes/progress); its sidebar uses `masterclassNav`.

**Mode B feature surface (Phase 2):** Shared helpers live in `src/lib/masterclass.ts` — `cohortMeta` (label/short/color/badge per age group; **always source cohort colors/labels from here**), the 12-week `curriculum` template, `isJoinableMC()` (10-min join gate — distinct from Mode A's 15-min `isJoinable()` in `time.ts`), and `buildSchedule()` (turns a first-session datetime + meeting link + cohort ids into 12 weekly insert rows). Teacher admin lives under `/teacher/masterclass*` (hub `TeacherMasterclass`, `…/cohorts` roster with create-cohort + CSV export, `…/sessions` with the generator + per-session CRUD using modal forms like `Sessions.tsx`) and is reached from `teacherNav`. Students get `/masterclass` (hub: next-session countdown + 10-min Join gate), `/masterclass/sessions` (list), `/masterclass/sessions/:id` (detail — shows only the activity for the viewer's own `age_group`). Session visibility is enforced by the `current_cohort_id() = ANY(cohort_ids)` RLS, so a plain `select('*')` from `masterclass_sessions` already returns only the student's cohort. Cohorts can be pre-created by the teacher or auto-created by `complete_onboarding`; a session with empty `cohort_ids` is hidden from all students (the teacher UI warns about this). Mode B notifications are email-first (Phase 4) so masterclass writes don't call `notify.ts`.

**Mode B quizzes (Phase 3):** Types/helpers in `src/lib/masterclass.ts` — `quizTypeMeta`, `QuizQuestion` (`mcq` auto-scorable / `short` hand-graded), `maxScoreOf()` (sum of inline question points, else `max_score`), `autoScoreMcq()` (scores only the MCQ portion). Single-submission model per `UNIQUE(quiz_id,user_id)` on `masterclass_submissions`. Teacher: `/teacher/masterclass/quizzes` (list + per-quiz submission/pending counts), `…/quizzes/new` + `…/quizzes/:id/edit` (shared `TeacherMasterclassQuizBuilder` — inline question editor with MCQ/short toggle, options, correct-answer marking, points, up/down reorder [not literal drag-and-drop], **or** a file attachment uploaded to `masterclass-materials`), `…/quizzes/:id/grade` (`TeacherMasterclassQuizGrade` — shows answers + auto-MCQ suggestion, teacher sets final score + feedback → `status='graded'`). Student: `/masterclass/quizzes` (filterable pending/submitted/graded) + `/masterclass/quizzes/:id` (`MasterclassQuizAttempt` — inline answer form or file upload to `masterclass-submissions/<uid>/…`; once submitted it becomes a read-only result view, with score+feedback when graded). File up/downloads use the download-as-blob pattern. Quiz emails + due-date reminders + certificate-on-completion are Phase 4/5.

**Mode B progress + certificates (Phase 5):** `/masterclass/progress` (`MasterclassProgress`, in `masterclassNav`) calls `certificate_status()` for the rings (quizzes completed / sessions attended), a derived badge shelf, and the certificate card. "Sessions attended" is fed by `recordAttendance(sessionId)` (in `masterclass.ts`), fired on every Join click (idempotent upsert into `masterclass_attendance`). Certificate PDF is generated **client-side** with `@react-pdf/renderer`, which lives in `src/lib/certificate.tsx` and is **dynamically imported** (`await import('../lib/certificate')`) so it stays in its own ~1.4 MB `certificate-*.js` chunk out of the main bundle — never static-import it. Flow on download: generate blob → upload to `certificates/<uid>/<cohort_id>.pdf` (upsert) → `issue_certificate(path)` re-validates eligibility and upserts the row → blob also downloaded locally. Eligibility = a submission exists for every quiz targeting the student's cohort (≥1 quiz). Certificate email (Template 8) is deferred to Phase 4.

**Mode B Prompt Quest (the "Prompt Arena"):** a game-style, **non-MCQ** challenge at `/masterclass/arena` (`MasterclassArena`, in `masterclassNav` — now one of the four mobile bottom-tab items: Home/Sessions/Quizzes/Prompt Quest). It tests the Week-2 5-ingredient prompt recipe (WHO/WHAT/DETAILS/OUTPUT/STYLE). UI is a **Quest Map** state machine (`map` → `play` → `reveal`): the map shows 3 gated nodes (easy→medium→hard) on a path; play shows the learner's **unique scenario** (claimed via `claim_arena_topic`) + a recipe cheat sheet + a prompt editor; reveal animates five ingredient bars to their 0-2 sub-scores, counts up the /10 total, pops stars, and fires a self-contained framer-motion **confetti** burst (no new dep) for scores ≥8. Grading is an **AI judge** — `gradeArenaPrompt()` invokes the `grade_prompt` Edge Function (Gemini), which is the only way a score is written (learners have no write policy). All arena types/helpers live in `src/lib/masterclass.ts` (`ARENA_INGREDIENTS`, `ARENA_ROUNDS`, `fetchArenaState`/`claimArenaTopic`/`gradeArenaPrompt`, `arenaVerdict`). Points fold into the shared `masterclass_leaderboard()` (quiz + quest XP on one board). Teacher results dashboard: `/teacher/masterclass/arena` (`TeacherMasterclassArena`, in `teacherMasterclassNav`) — per-learner easy/medium/hard score matrix with expandable prompt + breakdown, cohort filter, pool stats. Topics are seeded server-side; "add more scenarios" = INSERT into `arena_topics` (claim raises a clean error if a cohort exhausts a round's unique pool).

**Email pipeline (Phase 4) — FLAGGED OFF.** Not built: it needs an email transport credential (Resend or SMTP) which isn't configured yet. **Supabase's own email only sends auth emails (signup/recovery) — it cannot be reused for custom transactional sends.** `src/lib/email.ts` is the single intentional integration point: `EMAIL_PIPELINE_ENABLED = false` + a no-op `queueEmail(template, recipientId, payload, sendAt?)`. Real enqueuing must be server-side (the `email_queue` table has no client INSERT policy by design); Phase 4 adds the `dispatch_emails` Edge Function + Postgres enqueue triggers + the 9 React Email templates + `/unsubscribe`. Mode B masterclass flows deliberately don't call `notify.ts` (that's Mode A's bell system).

**Mode B auth surface (Phase 1):** `/join` (public) → `signInWithOAuth({provider:'google', redirectTo: origin + '/auth/callback'})` → `/auth/callback` polls for the session then routes by role + `onboarding_complete` (incomplete `student_group` → `/onboarding`, else `roleHome`). The `student_group` profile row is created synchronously by the `handle_new_user()` trigger, so it exists by the time the callback's session does. `/onboarding` (gated by `onboardingGuard`: only an unfinished `student_group`) is a 2-step name + age-group form that calls `complete_onboarding` and enters `/masterclass`. `groupGuard` forces unfinished `student_group` users back to `/onboarding`. `/setup` (public) is the lock-aware teacher bootstrap — shows "already complete" when `teacher_exists()` is true (the live state), otherwise a one-time create form using `bootstrap_teacher`. Requires the Google provider enabled in Supabase Auth with `<origin>/auth/callback` (prod + `http://localhost:5173/auth/callback` dev) in the redirect allow-list.

Routes are duplicated by role prefix (`/teacher/*` and `/student/*`) and reuse the same page components (`Courses`, `Sessions`, `Documents`, `Quizzes`, `Slides`, `SlideViewer`, `Notifications`). Pages are expected to read `role` from the store and conditionally render teacher-only controls.

### State
Zustand store at `src/store/useAppStore.ts` holds `user`, `profile`, `role`, `isLoading`. `profile` is currently typed `any` — when adding generated DB types, replace this. Setting profile auto-derives role, so always go through `setProfile`. The store also **caches profile in `localStorage` under `educonnect:profile-cache`** and hydrates it synchronously at module load — this is what makes a returning user's dashboard render instantly instead of flashing a skeleton. `setProfile` writes to localStorage on every call; `setProfile(null)` clears it.

Server state is owned by **TanStack Query** (`@tanstack/react-query`). Dashboards (`TeacherDashboard.tsx`, `StudentDashboard.tsx`) and the quiz submitter-profile lookup use queries; gate skeleton screens on `isPending` rather than ad-hoc `useEffect` loading flags.

### Shared libs
- `src/lib/nav.ts` — `teacherNav` / `studentNav` arrays + `navForRole(role)`. Always import from here, do not duplicate nav arrays in pages.
- `src/lib/time.ts` — `formatPKT`, `formatCET` (uses `Intl.DateTimeFormat` with `Asia/Karachi` and `Europe/Copenhagen`), `useCountdown(target)` hook ticking once per second, `isJoinable(scheduled_at, duration_min)` for the 15-min Zoom gate.
- `src/lib/notify.ts` — `notifyStudents(type, title, body, related_id)` and `notifyTeachers(...)`. Sessions/Documents/Quizzes call these on inserts/updates so the bell + Notifications page light up via realtime.

### Supabase client
Single client in `src/lib/supabase.ts`, reads `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` from `frontend/.env`. Falls back to localhost dummy values if env vars are missing — auth will silently fail in that state. **Symptom: login button does nothing, or the app sits in an infinite loading spinner after sign-in.** Always verify `frontend/.env` first when debugging login.

### RLS model (important for any DB query)
All tables have RLS enabled. The current policies (see migration) grant **students blanket SELECT** across `semesters`, `courses`, `sessions`, `documents`, `quizzes` (no per-enrollment scoping yet) and **teachers full CRUD** via an `EXISTS` check on their own `profiles` row. `notifications` is properly scoped to `recipient_id = auth.uid()`. When adding queries, do not assume student row-level isolation beyond `notifications` — the PRD describes per-enrollment scoping that is not yet implemented in SQL.

### Layout / UI system
- `components/DashboardLayout.tsx` wraps all dashboard pages with `Sidebar` + `TopBar`. Pass `title` and a `navItems` array per role. The shell is **mobile-first**: sidebar is an off-canvas drawer below `lg`, content drops the `ml-60` offset (`lg:ml-60`), and `TopBar` shows a hamburger that toggles `sidebarOpen` state owned by `DashboardLayout`. Heights use `100dvh` (not `100vh`) for iOS Safari; safe-area insets apply via `env(safe-area-inset-*)`. **Mode B exception (PRD §8.5):** when `role === 'student_group'`, `DashboardLayout` renders `components/BottomTabBar.tsx` on mobile instead of the drawer (and passes `onMenuClick={undefined}` so `TopBar` hides the hamburger; `TopBar` only renders the hamburger when `onMenuClick` is set). ### File downloads from Supabase Storage
**Never** wire a `<a href={signedUrl} download>` for Storage downloads. Two reasons: (1) Supabase signed URLs default to `Content-Disposition: attachment` and can mangle HTML rendering; (2) the `download` attribute is ignored on cross-origin URLs, so the browser navigates to the file instead of saving it. Always use the **download-as-blob** pattern: `supabase.storage.from(bucket).download(path)` → wrap the bytes in a `Blob` (set `type: 'text/html;charset=utf-8'` for HTML decks to fix mojibake) → `URL.createObjectURL(blob)` → either set on iframe `src` (with `sandbox="allow-scripts allow-same-origin"`) or trigger an anchor click + `revokeObjectURL`. See `pages/SlideViewer.tsx`, `pages/Documents.tsx`, `pages/CourseDetail.tsx`, `pages/Quizzes.tsx` for the canonical implementation. When uploading HTML, set `contentType: 'text/html; charset=utf-8'` so the bytes round-trip cleanly.

### `html2tsx.js`
Quick-and-dirty converter that wraps raw HTML mockups into React components in `src/pages/`. The source HTML lives at `frontend/src/Login.html`, `frontend/src/TeacherDashboard.html`, `frontend/src/StudentDashboard.html`. ## Conventions to preserve

- Time storage: `scheduled_at` is `timestamptz` in UTC. UI must render both PKT (teacher) and CET/CEST (student) — see PRD §5.4 / §9.
- Account creation: public signup is disabled. Teacher creates students via Supabase Admin API or dashboard; the app never exposes a signup form.
- **First-teacher bootstrap:** after creating the auth user in Supabase Dashboard → Authentication, you must also insert the matching `profiles` row with `role='teacher'`. Without it, login redirects loop indefinitely because `App.tsx` gates every protected route on both `user` AND `role`. Example: `INSERT INTO profiles (id, role, full_name, email) VALUES ('<auth-user-uuid>', 'teacher', 'Dr. Aris', 'teacher@school.edu');`
- File uploads go through Supabase Storage with private buckets + signed URLs (`documents`, `quiz-submissions`, `slides`). The one exception is `branding`, which is public on purpose — it's the only way email clients can fetch the logo. Don't add other public buckets.
- Quiz submissions use a **single-row model**: the `quizzes` table itself stores `submitted_by` / `submitted_at` / `submitted_filename`, since this product is one-teacher / one-student. Don't add a separate submissions table without first reconciling with the PRD.
- AI Feed news is **fully automated** — the Edge Function publishes directly to `feed_items.status = 'published'` without a teacher review step. Cover images are proxied into our own `branding` bucket (under `feed-covers/`) so third-party hot-link blocks don't break the UI. The only acceptable use of the public `branding` bucket beyond the logo is feed covers; private buckets are still the rule everywhere else.
